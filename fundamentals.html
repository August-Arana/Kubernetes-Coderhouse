Before You Dive In
Core Kubernetes Concepts Used Here
Namespace
A virtual cluster inside the real cluster. Isolates resources so names don't collide.
Secret
Stores sensitive data (passwords, tokens) base64-encoded, separately from code.
ConfigMap
Stores non-sensitive config (ports, hostnames) as plain key-value pairs.
Deployment
Manages stateless app replicas. Handles rolling updates and self-healing automatically.
StatefulSet
Like a Deployment but for stateful apps. Gives each pod a stable identity and ordered startup.
Service
A stable network endpoint (DNS name + port) that routes traffic to a set of pods via labels.
PersistentVolumeClaim
A request for disk storage. K8s fulfills it with a PersistentVolume from the infrastructure.
Liveness Probe
K8s checks this to know if a container is alive. Failure → container is killed and restarted.
Readiness Probe
K8s checks this before sending traffic. Failure → pod is removed from the load balancer.
Labels & Selectors
The glue of Kubernetes. Labels tag pods; selectors on Services and Deployments find them.
Resource Limits
CPU/memory guardrails per container. Requests = minimum needed. Limits = hard ceiling.
TopologySpread
Tells the scheduler to spread pods across nodes (or zones) for high availability.
Layer 1 Infrastructure — Namespace · Secret · ConfigMap
01
01-namespace.yml
Namespace Isolation
YAML
apiVersion: v1
kind: Namespace
metadata:
  name: innovate-ecommerce

What it does

Creates a virtual cluster called innovate-ecommerce inside your Kubernetes cluster. Every other resource in this project lives inside this namespace.

Why it exists

Without a namespace, everything lands in default. Namespaces let multiple apps, teams, or environments share the same cluster without interfering with each other. Resource names only need to be unique within a namespace — you could have a postgres-db in innovate-ecommerce and another in shop-v2 with no conflict.

Key fields

apiVersion: v1Every manifest declares the API version. v1 is the core Kubernetes API — the most stable group, used for pods, services, secrets, etc.
kind: NamespaceTells Kubernetes what resource to create. The kind determines which schema rules apply.
metadata.nameThe namespace identifier. Every other manifest in this project references it with namespace: innovate-ecommerce.
In the Cloud

In EKS, GKE, or AKS, namespaces are identical — same YAML, same concept. Common real-world patterns: one namespace per environment (staging, production), or one per team. You can attach ResourceQuotas to a namespace to cap CPU/memory usage per team, and NetworkPolicies to control which namespaces can talk to each other.

02
02-secret.yml
Secret Configuration Security
YAML
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
  namespace: innovate-ecommerce
type: Opaque
data:
  POSTGRES_USER: cG9zdGdyZXM=  #postgres
  POSTGRES_PASSWORD: c2VjcmV0  #secret
  POSTGRES_DB: ZWNvbW1lcmNl  #ecommerce

What it does

Stores the database credentials separately from your application code. Other resources can reference specific keys from this Secret and Kubernetes injects them as environment variables into containers at runtime.

Why it exists

Hard-coding passwords in Dockerfiles or application code creates security risks and makes rotation painful. Secrets decouple credentials from code — you can update a password without rebuilding images or changing deployments.

Key fields

type: OpaqueThe generic Secret type. K8s also has built-in types like kubernetes.io/tls for certificates or kubernetes.io/dockerconfigjson for registry auth. Opaque means "user-defined, arbitrary data."
data:All values here must be base64-encoded. Generate with: echo -n "secret" | base64. The decoded value is what the container receives.
POSTGRES_USER, etc.The keys are the env var names that other manifests reference with secretKeyRef.key: POSTGRES_USER.
Critical Gotcha

base64 is encoding, not encryption. Anyone with kubectl get secret db-credentials -o yaml access can trivially decode all values. In this project it's a minikube teaching setup — fine. In production, never commit Secret YAMLs to git with real passwords. The file is shown here for learning purposes only.

In the Cloud

Real production setups use the External Secrets Operator: it connects K8s to AWS Parameter Store / Secrets Manager, GCP Secret Manager, or HashiCorp Vault. Your manifest just says "pull the secret named X from the vault" — actual values never live in YAML files or git. Secrets are also encrypted at rest in managed K8s clusters (EKS, GKE) via KMS integration.

03
03-configmap.yml
ConfigMap Configuration
YAML
apiVersion: v1
kind: ConfigMap
metadata:
  name: api-config
  namespace: innovate-ecommerce
data:
  PORT: "3000"
  DATABASE_HOST: postgres-db-service
  DATABASE_PORT: "5432"

What it does

Stores non-sensitive configuration as plain text key-value pairs. The backend deployment mounts this entire ConfigMap as environment variables using envFrom, so every key becomes a variable available inside the container.

Why it exists

The same Docker image can run in dev, staging, and production with different ConfigMaps — no need to rebuild the image when a hostname or port changes. ConfigMap = config that changes per environment; Secret = config that is sensitive.

Key fields

data:Plain text — no encoding required. Numbers must be quoted ("3000") because YAML would otherwise parse them as integers, but K8s expects strings.
DATABASE_HOSTPoints to postgres-db-service — the exact name of the ClusterIP service in file 05. Kubernetes DNS resolves this name to the service's virtual IP automatically.
PORTThe port the backend Express server will listen on. Injected via envFrom, so process.env.PORT returns "3000" inside the container.
Note: Env var naming

The backend's server.js reads process.env.PGHOST, not DATABASE_HOST. So in this project, DATABASE_HOST is injected but not actually consumed — the server falls back to its hardcoded default 'postgres-db-service'. Both resolve to the same address, so it works, but it's a naming inconsistency worth knowing. In a real app, align the ConfigMap keys with what the code reads.

In the Cloud

ConfigMaps are identical across cloud providers — same YAML. With Helm (the standard deployment tool), ConfigMap values come from values.yaml files per environment. Some teams replace ConfigMaps entirely with cloud-native config services (AWS AppConfig, GCP Runtime Config), but plain ConfigMaps remain the most portable approach.

Layer 2 Database — Two Services + StatefulSet (files 04 – 06)
04
04-postgres-service-headless.yml
Service Headless StatefulSet DNS
YAML
apiVersion: v1
kind: Service
metadata:
  name: postgres-db-headless
  namespace: innovate-ecommerce
spec:
  clusterIP: None
  selector:
    app: postgres-db
  ports:
  - port: 5432
    targetPort: 5432

What it does

Creates a headless service — a service with no virtual IP (clusterIP: None). Instead of load-balancing, Kubernetes uses it purely for DNS: each StatefulSet pod gets a unique DNS entry like postgres-db-0.postgres-db-headless.innovate-ecommerce.svc.cluster.local.

Why it exists — and why two Postgres services?

StatefulSets require a headless service, declared in their serviceName field. This is how Kubernetes assigns each pod a stable, predictable identity. Without it, pods would get random names and the StatefulSet can't guarantee ordered operations.

You need two services for Postgres: this headless one (for stable pod identity) and the ClusterIP in file 05 (for application connection strings). They serve different purposes.

Key fields

clusterIP: NoneThis single line is what makes the service headless. Without a virtual IP, DNS queries return the actual pod IPs directly instead of a load-balancer IP.
selector.app: postgres-dbMatches pods that have the label app: postgres-db. The StatefulSet template defines this label on its pods.
In the Cloud

Same concept applies in EKS/GKE/AKS. The headless service pattern is needed any time you run a StatefulSet — Kafka, Cassandra, Zookeeper, Redis Cluster, etc. In practice, teams running databases in cloud Kubernetes still need both services, just like here. More commonly, the database is moved out entirely to a managed service (see file 06).

05
05-postgres-service-clusterip.yml
Service ClusterIP Internal DNS
YAML
apiVersion: v1
kind: Service
metadata:
  name: postgres-db-service
  namespace: innovate-ecommerce
spec:
  selector:
    app: postgres-db
  ports:
  - port: 5432
    targetPort: 5432

What it does

Creates a stable virtual IP and DNS name (postgres-db-service) that routes traffic to whichever Postgres pod is running. This is a standard ClusterIP service — internal-only, not reachable from outside the cluster.

Why it exists

The backend's server.js connects to postgres-db-service as the hostname. Services decouple the consumer (backend) from the exact pod location. If the Postgres pod restarts on a different node, the service still routes correctly — the backend never needs to know or care.

Notice there's no type: field here. When omitted, Kubernetes defaults to ClusterIP — a virtual IP that exists only within the cluster's private network.

Key fields

name: postgres-db-serviceThis exact string becomes the DNS hostname. From any pod in the same namespace: psql -h postgres-db-service works. K8s DNS auto-creates the entry.
selector.app: postgres-dbK8s continuously watches for pods with this label and keeps the service's endpoint list updated. This is the label+selector wiring that connects all Kubernetes resources.
port / targetPortport is what callers connect to; targetPort is the container's actual port. Here they're both 5432, but they can differ (e.g. service on 80, container on 8080).
In the Cloud

ClusterIP services are identical across cloud providers. In production, this entire layer (files 04–06) is often replaced by a managed database — AWS RDS, GCP Cloud SQL, Azure Database for PostgreSQL. The managed DB gives you automatic failover, backups, and scaling. Your backend then connects to the managed DB's hostname directly — no ClusterIP service needed for the DB itself.

06
06-postgres-statefulset.yml
StatefulSet PVC Health Probes Resource Limits
YAML
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres-db
  namespace: innovate-ecommerce
spec:
  serviceName: postgres-db-headless
  replicas: 1
  selector:
    matchLabels:
      app: postgres-db
  template:
    metadata:
      labels:
        app: postgres-db
    spec:
      containers:
      - name: postgres
        image: postgres:15
        ports:
        - containerPort: 5432
        env:
        - name: POSTGRES_USER
          valueFrom:
            secretKeyRef:
              name: db-credentials
              key: POSTGRES_USER
        - name: POSTGRES_PASSWORD
          valueFrom:
            secretKeyRef:
              name: db-credentials
              key: POSTGRES_PASSWORD
        - name: POSTGRES_DB
          valueFrom:
            secretKeyRef:
              name: db-credentials
              key: POSTGRES_DB
        volumeMounts:
        - name: pgdata
          mountPath: /var/lib/postgresql/data
        resources:
          requests:
            memory: "512Mi"
            cpu: "250m"
          limits:
            memory: "1Gi"
            cpu: "500m"
        livenessProbe:
          exec:
            command: ["pg_isready"]
          initialDelaySeconds: 30
          periodSeconds: 10
        readinessProbe:
          exec:
            command: ["pg_isready"]
          initialDelaySeconds: 10
          periodSeconds: 5
  volumeClaimTemplates:
  - metadata:
      name: pgdata
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests:
          storage: 1Gi

What it does

Runs PostgreSQL as a StatefulSet — the right tool for databases and any app that needs a stable identity across restarts. Manages one pod (postgres-db-0), persistent disk, secrets injection, and health monitoring.

StatefulSet vs Deployment

  • Deployment pods get random names (backend-api-xyz-abc), can be replaced in any order, and share nothing between replicas.
  • StatefulSet pods get stable ordinal names (postgres-db-0, postgres-db-1…), start and stop in order, and each gets its own dedicated PersistentVolume.

Databases need stable identity because data must stay with a specific pod instance. replicas: 1 here — PostgreSQL HA (high availability with replicas) is complex and usually handled by cloud-managed DBs.

Key fields

serviceNameLinks to the headless service (file 04). Required by all StatefulSets — this is how K8s constructs the stable DNS names for each pod.
selector.matchLabelsThe StatefulSet manages pods that carry this label. The pod template also sets this label — they must match exactly or K8s rejects the manifest.
secretKeyRefPulls individual keys from the Secret (file 02) and injects them as env vars. The postgres:15 image reads POSTGRES_USER, POSTGRES_PASSWORD, POSTGRES_DB on first boot to initialize the database.
resources.requestsThe minimum guaranteed resources. The scheduler uses this to find a node with enough capacity. 250m CPU = 0.25 of one CPU core.
resources.limitsHard ceiling. The container is OOMKilled if it exceeds memory, or throttled if it exceeds CPU. requests ≤ limits always.
livenessProbeRuns pg_isready every 10s. If it fails, K8s restarts the container. Starts checking after 30s (DB needs time to initialize).
readinessProbeSame check but starts after 10s. While failing, the pod is removed from all service endpoints — no traffic until Postgres is genuinely accepting connections.
volumeClaimTemplatesThe unique feature of StatefulSets. Each pod gets its own PVC automatically named pgdata-postgres-db-0. Delete and recreate the pod — same disk reattaches.
mountPath/var/lib/postgresql/data is where Postgres writes its data files. The PVC is mounted here so data survives pod restarts.
In the Cloud

In production, replace the entire DB layer (files 04–06) with a managed database. AWS RDS, GCP Cloud SQL, or Azure Database for PostgreSQL give you automatic backups, failover, read replicas, and zero maintenance. Your backend just points to the managed endpoint. The PVC accessMode: ReadWriteOnce maps to cloud storage classes like gp3 (AWS) or pd-ssd (GCP) when running real K8s.

    matchLabels:
      app: backend-api

            name: api-config
        env:
        - name: POSTGRES_USER
          valueFrom:
            secretKeyRef:
              name: db-credentials
              key: POSTGRES_USER
        - name: POSTGRES_PASSWORD
          valueFrom:
            secretKeyRef:
              name: db-credentials
              key: POSTGRES_PASSWORD
        - name: POSTGRES_DB
          valueFrom:
            secretKeyRef:
              name: db-credentials
              key: POSTGRES_DB
        resources:
          requests:
            memory: "256Mi"
            cpu: "200m"
          limits:
            memory: "512Mi"
            cpu: "400m"
        livenessProbe:
          httpGet:
            path: /healthz
            port: 3000
          initialDelaySeconds: 15
          periodSeconds: 10
        readinessProbe:
          httpGet:
            path: /readyz
            port: 3000
          initialDelaySeconds: 5
          periodSeconds: 5
      topologySpreadConstraints:
      - maxSkew: 1
        topologyKey: kubernetes.io/hostname
        whenUnsatisfiable: ScheduleAnyway
        labelSelector:
          matchLabels:
            app: backend-api

What it does

Runs 4 identical copies of the Node.js API, managed by a Deployment. If any pod crashes, the Deployment controller automatically replaces it. If you update the image, it does a rolling update — replacing pods one-by-one with zero downtime.

The labels and selectors chain

The Deployment's selector.matchLabels must exactly match the template.metadata.labels it creates. This is how K8s knows which pods belong to which Deployment. The Service in file 08 uses the same label to know where to send traffic.

Key fields

replicas: 4The Deployment ensures exactly 4 pods are running at all times. Scale up: change to 8 and kubectl apply again. Scale down: change to 2. No downtime.
imagePullPolicy: Neverminikube-specific. Tells K8s to never try to pull from a registry — use whatever image already exists in the local Docker daemon. You build the image inside minikube's Docker context first. Remove this in cloud.
envFrom: configMapRefInjects all keys from the api-config ConfigMap as environment variables. Cleaner than listing each key individually when you want the whole ConfigMap.
env: secretKeyRefInjects specific keys from the Secret. Unlike envFrom, this is selective — pull only the keys you need, with whatever env var name you choose.
livenessProbe /healthzHTTP GET to port 3000/healthz every 10s. The endpoint just returns 200 OK — it only checks if the Node.js process is alive. Failure → container restart.
readinessProbe /readyzHTTP GET to port 3000/readyz every 5s. This endpoint actually queries the database (SELECT 1). Failure → pod removed from load balancer. This prevents routing traffic to a backend that can't reach the DB.
topologySpreadConstraintsTells the scheduler: spread these pods across nodes, with no more than 1 pod difference per node (maxSkew: 1). ScheduleAnyway = soft constraint (schedule even if it can't satisfy perfectly). Prevents all 4 pods landing on 1 node.
In the Cloud

Remove imagePullPolicy: Never and set image to a full registry path like 123456789.dkr.ecr.us-east-1.amazonaws.com/backend-app:v1.2.3. K8s will pull from ECR (AWS), GCR/Artifact Registry (GCP), or ACR (Azure). Change topologyKey from kubernetes.io/hostname to topology.kubernetes.io/zone to spread pods across availability zones instead of just nodes — this is true high availability.

08-backend-service.yml
Service ClusterIP Internal Load Balancer
YAML
apiVersion: v1
kind: Service
metadata:
  name: backend-api-service
  namespace: innovate-ecommerce
spec:
  selector:
    app: backend-api
  ports:
  - port: 3000
    targetPort: 3000

What it does

Creates a stable internal DNS name and load balancer for the 4 backend pods. Any pod in the cluster can reach the API at http://backend-api-service:3000. K8s distributes requests across all healthy, ready backend pods in round-robin.

The name matters

The service name backend-api-service is hardcoded in the frontend deployment (file 09) as API_URL: http://backend-api-service:3000. If you rename this service, you must update the frontend env var too. This name-based wiring is how microservices find each other in Kubernetes.

Key fields

selector.app: backend-apiMatches pods from the Deployment in file 07. K8s dynamically updates the endpoint list as pods come and go — a pod that fails its readiness probe is automatically removed from routing.
port: 3000The port the service listens on. Callers use this port. Since both port and targetPort are 3000, the service just forwards directly to the container port.
In the Cloud

Same ClusterIP type works identically on all cloud K8s. For internal service-to-service communication this is always the right choice. If you needed the backend to be accessible from outside the cluster (e.g., for a mobile app), you'd add type: LoadBalancer — the cloud would provision a real load balancer with a public IP automatically.

What it does

Runs 4 replicas of the PHP/Apache frontend. Structurally identical to the backend Deployment — same concepts: replicas, labels, probes, topology constraints. Lower resource limits because serving a PHP page is cheaper than running a DB-connected API.

How the frontend finds the backend

The API_URL environment variable is set to http://backend-api-service:3000. Inside index.php, PHP uses cURL to call this URL. The Kubernetes DNS resolver converts backend-api-service to the ClusterIP service's virtual IP (file 08), which then routes to one of the 4 backend pods.

Key fields

env.API_URLDirect value injection (no ConfigMap/Secret reference). In a more production-ready setup, this would come from a ConfigMap so you can change the backend URL without editing the Deployment YAML.
livenessProbe: /HTTP GET to root path. Unlike the backend, the frontend just checks if Apache is serving — there's no separate health endpoint. Any 2xx response means it's alive.
resourcesLower limits than backend: 128Mi–256Mi memory, 100m–200m CPU. Frontend is mostly I/O — accepting requests and passing through to the API — so it needs less compute than the API layer.
In the Cloud

In cloud, the frontend would typically be behind an Ingress (a layer-7 HTTP router), not a NodePort. The Ingress handles TLS termination (HTTPS), domain routing, and path-based routing. The frontend pods stay exactly like this; only how external traffic enters the cluster changes.

10-frontend-deployment.yml (misnamed — this is the frontend Service)
Service NodePort External Access
YAML
apiVersion: v1
kind: Service
metadata:
  name: frontend-service
  namespace: innovate-ecommerce
spec:
  selector:
    app: frontend-app
  ports:
  - port: 80
    targetPort: 80
  type: NodePort

What it does

Exposes the frontend to traffic from outside the cluster. A NodePort service opens a port on every cluster node (minikube assigns one in the 30000–32767 range). You reach the app via the node's IP + that port.

The three Service types explained

  • ClusterIP (default) — internal only. Only reachable from within the cluster. Used for Postgres (file 05) and backend (file 08).
  • NodePort — opens a port on every node. Reachable from outside via node-ip:nodePort. Used here for the frontend. Only appropriate for development and minikube.
  • LoadBalancer — cloud-native. Provisions a cloud load balancer with a public IP. The production equivalent of NodePort.

Key fields

type: NodePortThis one field changes the entire access model. Without it, the service defaults to ClusterIP (internal only).
selector.app: frontend-appRoutes to the pods created by the Deployment in file 09. Same label-based wiring as everywhere else.
port: 80The service port. NodePort also allocates a random high port (e.g., 31234) on the node. In minikube, use: minikube service frontend-service -n innovate-ecommerce — it opens the URL automatically.
In the Cloud

Replace type: NodePort with type: LoadBalancer and your cloud provider automatically creates a load balancer and assigns a public IP. Even better: use an Ingress resource with cert-manager for automatic TLS. One Ingress can route traffic to many services based on hostname or path, replacing multiple LoadBalancer services and saving on cloud load balancer costs.

File Naming Note

This file is named 10-frontend-deployment.yml but contains a Service, not a Deployment. The Deployment is in file 09. In a real project, name it 10-frontend-service.yml — the number sequence is the important part (apply order), but the name should describe the content.

Applications Dockerfiles & Source Code — What runs inside the pods
Backend Application
backend/ — Node.js + Express

The backend is a REST API that connects to PostgreSQL and exposes CRUD endpoints for products. It is built into a Docker image called backend-app and runs in the pods managed by the Deployment in file 07.

backend/Dockerfile
FROM node:20
WORKDIR /app
COPY package.json .
RUN npm install
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]

Layer caching trick: COPY package.json . + RUN npm install happens before COPY . .. Docker caches each layer. If you only change server.js, Docker skips the npm install layer — rebuilds are much faster. Only when package.json changes does it reinstall dependencies.

backend/server.js — Health Endpoints
// Liveness: just checks if the process is alive
app.get('/healthz', (req, res) => {
  res.status(200).send('OK');
});

// Readiness: checks if the DB connection works
app.get('/readyz', async (req, res) => {
  try {
    await pool.query('SELECT 1');
    res.status(200).send('Ready');
  } catch (err) {
    res.status(503).send('Not Ready: DB connection failed');
  }
});

Why two health endpoints? /healthz (liveness) is cheap — it just confirms the Node.js process hasn't crashed. /readyz (readiness) is heavier — it actually queries the database. Kubernetes uses them for different decisions: restart (liveness failure) vs. remove from load balancer (readiness failure). A backend that's alive but can't reach the DB should serve no traffic, but should not be restarted — hence two separate probes.

Connection Pool & Env Var Mismatch

The pg.Pool reads PGHOST, PGUSER, PGPASSWORD, PGDATABASE. But the Deployment injects POSTGRES_USER, POSTGRES_PASSWORD, POSTGRES_DB from the Secret. These names don't match, so the pg library uses its hardcoded defaults ('postgres-db-service', 'postgres', 'secret', 'ecommerce') which happen to be the same values. It works — but it's an accidental match. Fix: rename the env vars in the Deployment to PGHOST, PGUSER, PGPASSWORD, PGDATABASE so the intent is explicit.

Frontend Application
frontend/ — PHP + Apache
frontend/Dockerfile
FROM php:apache
RUN apt-get update && \
    apt-get install -y libpq-dev && \
    docker-php-ext-install mysqli && \
    docker-php-ext-enable mysqli
COPY index.php /var/www/html

Base image: php:apache bundles PHP + Apache in one image — zero config needed to serve PHP files. The RUN installs the MySQL/PostgreSQL C library and activates the mysqli extension. There's no CMD — Apache starts automatically (the base image's CMD handles it). index.php is placed where Apache looks for files by default.

frontend/index.php — API call pattern
$api_url = getenv('API_URL')
         ?: 'http://backend-api-service:3000';

// cURL helper — calls the backend REST API
function callApi($method, $path, $data=null) {
  global $api_url;
  $ch = curl_init($api_url . $path);
  curl_setopt($ch, CURLOPT_CUSTOMREQUEST, $method);
  curl_setopt($ch, CURLOPT_RETURNTRANSFER, true);
  if ($data) {
    curl_setopt($ch, CURLOPT_POSTFIELDS,
      json_encode($data));
  }
  $res = curl_exec($ch);
  return json_decode($res, true);
}

Server-side API calls: The PHP frontend doesn't make JavaScript fetch calls — it uses PHP cURL to call the backend API on the server side, then renders the HTML response. This is a "server-side rendering" pattern. The browser only sees final HTML — it never talks to the backend directly. All inter-pod communication is cluster-internal.

How to Run This
Deployment Order & Why It Matters

Files are numbered 01–10 because they have dependencies. The namespace must exist before anything else. Secrets and ConfigMaps before the pods that consume them. Services before the pods (so DNS resolves when the pod starts). Apply in order:

1

Build the Docker images inside minikube's Docker daemon

minikube has its own Docker daemon separate from your host. You must build images there so imagePullPolicy: Never can find them.

$ eval $(minikube docker-env) # point your shell at minikube's Docker $ docker build -t backend-app ./backend $ docker build -t frontend-app ./frontend
2

Apply all manifests in order

Apply every file in numerical order. The -n flag isn't needed if your manifests include the namespace field (which they do), but the namespace itself must be created first.

$ kubectl apply -f 01-namespace.yml $ kubectl apply -f 02-secret.yml $ kubectl apply -f 03-configmap.yml $ kubectl apply -f 04-postgres-service-headless.yml $ kubectl apply -f 05-postgres-service-clusterip.yml $ kubectl apply -f 06-postgres-statefulset.yml $ kubectl apply -f 07-backend-deployment.yml $ kubectl apply -f 08-backend-service.yml $ kubectl apply -f 09-frontend-deployment.yml $ kubectl apply -f 10-frontend-deployment.yml # Or apply everything at once: $ kubectl apply -f .
3

Create the products table

PostgreSQL starts empty — you need to create the table manually (or with an init script). Exec into the Postgres pod:

$ kubectl exec -it postgres-db-0 -n innovate-ecommerce -- psql -U postgres -d ecommerce ecommerce=# CREATE TABLE products ( id SERIAL PRIMARY KEY, name VARCHAR(255) NOT NULL, description TEXT, price NUMERIC(10,2) NOT NULL, updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP );
4

Open the frontend in a browser

$ minikube service frontend-service -n innovate-ecommerce # Opens http://<minikube-ip>:<nodePort> in your browser automatically
Reference
kubectl Commands for This Project

All commands target the innovate-ecommerce namespace. Add -n innovate-ecommerce to every command.

Inspect everything
$ kubectl get all -n innovate-ecommerce $ kubectl get pods -n innovate-ecommerce $ kubectl get services -n innovate-ecommerce $ kubectl describe pod <pod-name> -n innovate-ecommerce
Debug a deployment
$ kubectl logs <pod-name> -n innovate-ecommerce $ kubectl logs -l app=backend-api -n innovate-ecommerce $ kubectl rollout status deployment/backend-api -n innovate-ecommerce $ kubectl rollout undo deployment/backend-api -n innovate-ecommerce
Scale and update
$ kubectl scale deployment backend-api --replicas=6 -n innovate-ecommerce $ kubectl get configmap api-config -n innovate-ecommerce -o yaml $ kubectl get secret db-credentials -n innovate-ecommerce -o yaml
Tear it all down
$ kubectl delete namespace innovate-ecommerce # Deletes everything inside the namespace at once # PVCs are also deleted — data is lost
